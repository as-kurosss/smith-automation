# ADR-0002: Windows UI tools and process management

**Status:** Accepted
**Date:** 2026-07-02
**Deciders:** @kurosss

---

## Context

After implementing ToolRegistry and ElementSelector (ADR-0001), specific tools are needed for Windows interface automation and process management. The following need to be defined:

1. **Element search pattern** — how tools obtain UI elements.
2. **Text input** — the difference between input simulation (send_keys) and programmatic setting (ValuePattern).
3. **Process management** — starting, stopping, and inventorying processes.

---

## Decision 1: FindTool — live search pattern

### Problem

UI elements can become invalid (window closed, element recreated). Storing elements for a long time is risky.

### Solution

**FindTool** performs an on-the-fly element search using the provided `ElementSelector` and stores the `SafeUIElement` in `ExecutionContext` under the specified key.

All subsequent tools (ClickTool, InputTextTool, SetTextTool) accept either:
- `element_key` — a reference to an element in the context (previously found via FindTool), or
- a full selector (name, automation_id, control_type, class_name, pid) — for on-the-fly search.

```rust
// Example: find + click
let mut selector = ElementSelector::new()
    .name("Submit")
    .control_type("Button")
    .pid(1234);
let element = selector.find_from_desktop()?;
ctx.set("btn", ContextValue::Custom(Arc::new(SafeUIElement::new(element))));

// Later — click
click.execute(json!({ "element_key": "btn" }), ...)

// Or in a single command:
click.execute(json!({
    "name": "Submit",
    "control_type": "Button",
    "pid": 1234
}), ...)
```

### Consequences

- The element is searched on each request — up-to-dateness is guaranteed.
- If the element is searched on the fly each time — a small overhead on the UIA request.
- FindTool is only needed when the same element is used multiple times.
- All found elements — `SafeUIElement` with `Arc<UIElement>` — live as long as there is at least one reference in the context.

## Decision 2: InputTextTool — send_keys vs InputTextTool — ValuePattern

### Problem

There are two fundamentally different ways to input text into a UI element:

1. **SendKeys** — simulates key presses (slower, but triggers all handlers).
2. **ValuePattern** — programmatically sets the value (faster, but not all elements support it).

### Solution

Create two separate tools:

**InputTextTool** (`windows.input_text`):
- Uses `UIElement::send_keys()`.
- Focuses the element before input.
- Suitable for input fields that respond to keyboard events.

**SetTextTool** (`windows.set_text`):
- Uses `ValuePattern::set_value()`.
- Only works with elements that support ValuePattern (Edit, ComboBox).
- Faster, does not require focus.

### Consequences

- Two tools instead of one with a flag — clearer for LLM (JSON Schema).
- InputTextTool works everywhere, SetTextTool is faster on supported elements.
- The LLM agent selects the tool based on the task: for fast filling — SetTextTool, for triggering onChange handlers — InputTextTool.

## Decision 3: ProcessTool — start via std::process::Command, stop/list via taskkill/tasklist

### Problem

Process management requires:
- Starting with arguments and working directory.
- Stopping by PID or process name.
- Getting a list of running processes.

### Solution

**ProcessTool** (`windows.process`) with three actions:

|| Action | Implementation |
||--------|---------------|
|| `start` | `std::process::Command::new(...).spawn()`. Returns PID. |
|| `stop` | `taskkill /F /PID <pid>` or `taskkill /F /IM <name>`. |
|| `list` | `tasklist /FO CSV /NH` with CSV parsing. |

**Why not WinAPI directly:**
- `CreateProcess` / `TerminateProcess` / `EnumProcesses` require `windows-sys` and unsafe.
- `std::process::Command` provides cross-platform launching.
- `taskkill`/`tasklist` are stable Windows utilities that don't require FFI.
- Consistency: before transitioning to platform-specific calls, the tool already works.
- Optimization to WinAPI can be done later if performance or extended capabilities are needed.

### Consequences

- Process launching is cross-platform (works on Linux/macOS too, but the module is behind `cfg(windows)`).
- Stopping and listing are Windows-only (via taskkill/tasklist).
- Simplicity at the cost of depending on external utilities (taskkill, tasklist).
- The PID returned by `start` can be passed to ElementSelector or SetTextTool/InputTextTool for process binding.

---

## Consequences (summary)

1. **Live search** — FindTool searches for the element on each request, guaranteeing up-to-dateness.
2. **Two input tools** — InputTextTool (send_keys) and SetTextTool (ValuePattern) cover all scenarios.
3. **ProcessTool** — start via std, stop/list via built-in Windows utilities.
4. **PID flow** — ProcessTool → FindTool (by PID) → ClickTool/InputTextTool/SetTextTool — complete automation lifecycle.
