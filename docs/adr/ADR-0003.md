# ADR-0003: Security hardening and concurrency improvements from audit

**Status:** Accepted
**Date:** 2026-07-02
**Deciders:** @kurosss

---

## Context

During the security and architecture audit (02.07.2026), the following problems requiring architectural solutions were identified:

1. **Command injection** — The HTTP API allowed launching arbitrary executables via `windows.process start`.
2. **Mutex lock** — `Mutex<AppState>` blocked all requests (including read-only `/health` and `/tools`) for the duration of any tool execution.
3. **Error chain loss** — `PlatformError(String)` formatted errors into a string, losing `source` and `#[source]` context.
4. **Double COM initialization** — `ElementSelector::build_condition()` created a new `UIAutomation` on each call, and `find_from_desktop()` created another one.

---

## Decision 1: Command allowlist for process start

### Problem

`std::process::Command::new(cmd_str)` accepted an arbitrary executable name from an HTTP request. An attacker could launch any process on the system.

### Solution

Added the `is_command_allowed()` function, which checks the executable name against an allowlist of known Windows utilities:

```rust
fn is_command_allowed(cmd: &str) -> bool {
    let allowed: HashSet<&str> = HashSet::from_iter([
        "notepad.exe", "calc.exe", "mspaint.exe",
        "cmd.exe", "powershell.exe", "explorer.exe",
        "write.exe", "wordpad.exe",
    ]);
// Extract file name from path, support / and \
    let name = cmd.rsplit_once(...).map(...).unwrap_or(cmd);
    allowed.contains(name)
}
```

**Why not regex/blacklist:** allowlist eliminates false positives and guarantees that any new executable requires an explicit decision.

**When to expand:** As new automated applications are added — expand the allowlist.

**Related changes:**
- `action_stop` changed from `.output()` (waiting for taskkill to complete) to `.spawn()` (fire-and-forget) — the process doesn't need to wait for taskkill to finish.
- `action_list` removed — `tasklist` is not needed for the core automation workflow.

## Decision 2: RwLock instead of Mutex for AppState

### Problem

`Mutex<AppState>` held an exclusive lock for the entire duration of tool execution (including UIA wait time). Read-only endpoints `/health` and `/tools` also had to wait in line.

### Solution

Replaced `tokio::sync::Mutex` with `tokio::sync::RwLock`:

|| Endpoint | Lock type | Reason |
||----------|-----------|--------|
|| `GET /health` | `read().await` | Only reading static data |
|| `GET /tools`  | `read().await` | Only reading tool list |
|| `POST /execute` | `write().await` | Mutating ExecutionContext |
|| `POST /reset` | `write().await` | Replacing ExecutionContext |

**Additionally:** `execute_handler` uses struct destructuring to minimize the lock scope — the lock is only held during `registry.execute()`, without response preparation.

## Decision 3: PlatformWithCause — preserving the error chain

### Problem

```rust
// Was — losing source:
SmithError::PlatformError(format!("Click failed: {e}"))
```

### Solution

New `PlatformWithCause` variant with `#[source]`:

```rust
#[error("Platform error: {message}")]
PlatformWithCause {
    message: String,
    #[source]
    source: Box<dyn Error + Send + Sync>,
}
```

All ~20 call sites in the `smith-windows` crate (click, find, input_text, set_text, process, selector) have been migrated to `PlatformWithCause`. This provides:

- Full error chain via `source()`.
- Compatibility with `anyhow::Context` and `thiserror`.
- Ability to log full backtrace via `tracing`.

---

## Decision 4: UIAutomation reuse in ElementSelector

### Problem

Each call to `build_condition()` created a new `UIAutomation` (COM init). And `find_from_desktop()` created another one. Two COM initializations per single element search.

### Solution

- `build_condition()` renamed to `build_condition_with(automation: &UIAutomation)`.
- `find_first()` and `find_all()` now accept a `&UIAutomation` parameter.
- `find_from_desktop()` creates one instance and passes it to all calls.
- Removed dead `timeout: Duration` — it was unused, only creating false expectations of retry logic.

---

## Consequences

1. **Command allowlist** — protection against injection at the cost of manually expanding the list when adding new tools.
2. **RwLock** — read-only endpoints are no longer blocked by long-running execute requests. Write-lock is held for minimal time.
3. **PlatformWithCause** — backward compatibility preserved (old `PlatformError` remains). All new errors use `PlatformWithCause`.
4. **UIAutomation reuse** — one COM initialization per search. Micro-optimization, but eliminates potential race conditions during parallel searches.
5. **Dead code** — timeout field and action_list removed.
6. **SafeUIElement creation** — moved inside `spawn_blocking` in find/input_text/set_text, since `UIElement` does not implement `Send`.
