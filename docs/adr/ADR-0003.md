# ADR-0003: Security hardening and concurrency improvements from audit

**Status:** Accepted
**Date:** 2026-07-02
**Deciders:** @kurosss

---

## Context

В ходе аудита безопасности и архитектуры (02.07.2026) были выявлены следующие проблемы, требующие архитектурных решений:

1. **Command injection** — HTTP API позволял запускать произвольные executables через `windows.process start`.
2. **Mutex-блокировка** — `Mutex<AppState>` блокировал все запросы (включая read-only `/health` и `/tools`) на время выполнения любого инструмента.
3. **Потеря цепочки ошибок** — `PlatformError(String)` форматировал ошибки в строку, теряя `source` и `#[source]` контекст.
4. **Двойная инициализация COM** — `ElementSelector::build_condition()` создавал новый `UIAutomation` на каждый вызов, и `find_from_desktop()` создавал ещё один.

---

## Decision 1: Command allowlist for process start

### Problem

`std::process::Command::new(cmd_str)` принимал произвольное имя исполняемого файла из HTTP-запроса. Злоумышленник мог запустить любой процесс на системе.

### Solution

Добавлена функция `is_command_allowed()`, проверяющая имя исполняемого файла по allowlist'у известных Windows-утилит:

```rust
fn is_command_allowed(cmd: &str) -> bool {
    let allowed: HashSet<&str> = HashSet::from_iter([
        "notepad.exe", "calc.exe", "mspaint.exe",
        "cmd.exe", "powershell.exe", "explorer.exe",
        "write.exe", "wordpad.exe",
    ]);
    // Извлекаем имя файла из пути, поддерживаем / и \
    let name = cmd.rsplit_once(...).map(...).unwrap_or(cmd);
    allowed.contains(name)
}
```

**Почему не regex/blacklist:** allowlist исключает ложные срабатывания и гарантирует что любой новый executable требует явного решения.

**Когда расширять:** По мере добавления новых автоматизируемых приложений — расширять allowlist.

**Сопутствующие изменения:**
- `action_stop` переведён с `.output()` (ожидание завершения taskkill) на `.spawn()` (fire-and-forget) — процессу не нужно ждать завершения taskkill.
- `action_list` удалён — `tasklist` не требуется для core automation workflow.

## Decision 2: RwLock вместо Mutex для AppState

### Problem

`Mutex<AppState>` удерживал эксклюзивную блокировку на всё время выполнения инструмента (включая ожидание UIA). Read-only эндпоинты `/health` и `/tools` тоже ждали очереди.

### Solution

Заменён `tokio::sync::Mutex` на `tokio::sync::RwLock`:

| Endpoint | Lock type | Почему |
|----------|-----------|--------|
| `GET /health` | `read().await` | Только чтение статических данных |
| `GET /tools`  | `read().await` | Только чтение списка инструментов |
| `POST /execute` | `write().await` | Мутация ExecutionContext |
| `POST /reset` | `write().await` | Замена ExecutionContext |

**Дополнительно:** `execute_handler` использует struct destructuring для минимизации области блокировки — lock удерживается только на время `registry.execute()`, без подготовки ответа.

## Decision 3: PlatformWithCause — сохранение цепочки ошибок

### Problem

```rust
// Было — потеря source:
SmithError::PlatformError(format!("Click failed: {e}"))
```

### Solution

Новый вариант `PlatformWithCause` с `#[source]`:

```rust
#[error("Platform error: {message}")]
PlatformWithCause {
    message: String,
    #[source]
    source: Box<dyn Error + Send + Sync>,
}
```

Все ~20 call sites в сrate `smith-windows` (click, find, input_text, set_text, process, selector) переведены на `PlatformWithCause`. Это даёт:

- Полную цепочку ошибок через `source()`.
- Совместимость с `anyhow::Context` и `thiserror`.
- Возможность логировать полный backtrace через `tracing`.

## Decision 4: UIAutomation reuse в ElementSelector

### Problem

Каждый вызов `build_condition()` создавал новый `UIAutomation` (COM init). И `find_from_desktop()` создавал ещё один. Две COM-инициализации на один поиск элемента.

### Solution

- `build_condition()` переименован в `build_condition_with(automation: &UIAutomation)`.
- `find_first()` и `find_all()` теперь принимают `&UIAutomation` параметр.
- `find_from_desktop()` создаёт один экземпляр и передаёт его во все вызовы.
- Удалён мёртвый `timeout: Duration` — не использовался, только создавал ложное ожидание retry-логики.

---

## Consequences

1. **Command allowlist** — защита от injection ценой ручного расширения списка при добавлении новых инструментов.
2. **RwLock** — read-only эндпоинты больше не блокируются долгими execute-запросами. Write-lock удерживается минимальное время.
3. **PlatformWithCause** — обратная совместимость сохранена (старый `PlatformError` остался). Все новые ошибки используют `PlatformWithCause`.
4. **UIAutomation reuse** — одна COM-инициализация на поиск. Микро-оптимизация, но устраняет потенциальные race conditions при параллельных поисках.
5. **Dead code** — timeout field и action_list удалены.
6. **SafeUIElement creation** — перенесён внутрь `spawn_blocking` в find/input_text/set_text, так как `UIElement` не реализует `Send`.
