# ADR-0004: Rust Canon v1 — typed tools, typestate context, structured errors

**Status:** Accepted
**Date:** 2026-07-08
**Deciders:** @kurosss

---

## Context

The Smith codebase had evolved organically: `Tool` trait used `Value` for input/output (no type safety), `ExecutionContext` was a plain struct (no compile-time state guarantees), error handling was a single enum with stringly-typed messages, and `unwrap()`/`expect()` calls were scattered through production code.

The ["Rust Canon" paper (arXiv 2602.11865)](https://arxiv.org/abs/2602.11865) describes a set of idiomatic Rust patterns for production systems. The goal was to retrofit these patterns onto Smith without changing external behaviour — all existing tests must pass, all tools must remain backward-compatible at the JSON boundary.

---

## Decision 1: Associated Types for Tool trait (§3.5)

### Problem

```rust
// Was — dynamic types, runtime errors on wrong JSON:
#[async_trait]
pub trait Tool: Send + Sync {
    fn name(&self) -> &'static str;
    fn description(&self) -> &'static str;
    fn schema(&self) -> Value;

    async fn execute(
        &self,
        config: Value,          // JSON — any shape, no compile-time check
        ctx: &mut ExecutionContext,
        token: CancellationToken,
    ) -> Result<Value, ToolError>; // JSON output — same issue
}
```

Input/output contracts were only documented in tests — nothing prevented passing `{"param": "string"}` to a tool expecting `{"param": 42}`.

### Solution

```rust
#[async_trait]
pub trait Tool: Send + Sync {
    type Input: DeserializeOwned + Serialize + Send;
    type Output: Serialize + Send;

    async fn execute(
        &self,
        input: Self::Input,       // Compiler-verified type
        ctx: &mut ExecutionContext,
        token: CancellationToken,
    ) -> Result<Self::Output, ToolError>;
}
```

Each tool declares its input/output types (e.g. `ClickInput`, `ClickOutput`), and mismatch is caught at compile time.

### Object-safe wrapper (DynTool)

Since `Tool` with associated types is not object-safe, a `DynTool` trait with JSON-based dispatch and a blanket impl for all `T: Tool` was created:

```rust
#[async_trait]
pub trait DynTool: Send + Sync {
    fn name(&self) -> &'static str;
    fn execute(&self, config: Value, ctx: &mut ExecutionContext, token: CancellationToken) -> Result<Value, ToolError>;
}

#[async_trait]
impl<T: Tool + Send + Sync> DynTool for T { ... }
```

`ToolRegistry` stores `Box<dyn DynTool>`, preserving dynamic dispatch while keeping the typed `Tool` trait for direct usage.

### Consequences

- **Positive:** Compile-time type safety for tool contracts. No more `serde_json::from_value` errors at runtime for parameter mismatches.
- **Positive:** Tool I/O is self-documenting — `ClickInput` fields are visible in the type, not buried in JSON schema.
- **Positive:** Zero-cost at the typed level — JSON serialisation only happens in `DynTool` bridge.
- **Trade-off:** Two traits (`Tool` + `DynTool`) instead of one. The blanket impl minimises boilerplate.

## Decision 2: Typestate for ExecutionContext (§2.3)

### Problem

`ExecutionContext` was a single struct — tools could be called before the context was initialised:

```rust
let mut ctx = ExecutionContext::new();
// ... some gap ...
registry.execute("windows.click", json!({}), &mut ctx, token).await;
// What if ctx wasn't properly initialised? No way to know at compile time.
```

### Solution

Introduced state markers with `PhantomData`:

```rust
pub struct Unvalidated;
pub struct Ready;

pub struct ExecutionContext<State = Ready> {
    scopes: Vec<HashMap<String, ContextValue>>,
    _state: PhantomData<State>,
}

impl ExecutionContext<Unvalidated> {
    pub fn new() -> Self { ... }
    pub fn validate(self) -> ExecutionContext<Ready> { ... }
}
```

`ToolRegistry::execute()` accepts only `&mut ExecutionContext<Ready>`:

```rust
pub async fn execute(
    &self,
    name: &str,
    config: Value,
    ctx: &mut ExecutionContext<Ready>,  // Compile-time: must be validated
    token: CancellationToken,
) -> Result<Value, ToolError>;
```

`I/O` methods (`get`, `set`, `push_scope`, `pop_scope`) are generic over any `State`.

### Consequences

- **Positive:** Impossible to call `ToolRegistry::execute()` on an unvalidated context — compiler error.
- **Positive:** The transition `Unvalidated → Ready` at `.validate()` is a single call site — easy to add real validation later (e.g. resource checks).
- **Positive:** Zero runtime cost — `PhantomData` is eliminated during compilation.
- **Trade-off:** Slight increase in generic boilerplate in code that stores or passes `ExecutionContext` without caring about state.

## Decision 3: Structured error reporting via thiserror (§4.2)

### Problem

```rust
// Was — stringly typed, no context, no source chain:
pub enum ToolError {
    InvalidInput(String),
    ElementNotFound { message: String },
    PlatformError(String),
    Timeout(u64),
}
```

All errors were treated uniformly — impossible to distinguish "element not found" from "invalid param" without string parsing. Source errors were discarded.

### Solution

```rust
#[derive(Error, Debug)]
pub enum ToolError {
    #[error("Invalid input: {message}")]
    InvalidInput {
        message: String,
        param: Option<String>,       // Which field caused the error
        input: Option<Value>,         // The bad input value
        timestamp: SystemTime,
    },

    #[error("Element not found: {message}")]
    ElementNotFound {
        message: String,
        selector: Option<Value>,     // What selector was used
        timestamp: SystemTime,
    },

    #[error("Operation cancelled")]
    Cancelled { timestamp: SystemTime },

    #[error("Platform error: {message}")]
    PlatformError {
        message: String,
        #[source] source: Box<dyn std::error::Error + Send + Sync>,
        input: Option<Value>,
        timestamp: SystemTime,
    },

    #[error("JSON error: {0}")]
    JsonError(#[from] serde_json::Error),
}
```

A `From<ToolError> for SmithError` conversion was added for backward compatibility with the existing error domain used by `smith-graph`.

### Consequences

- **Positive:** Every error carries timestamp and domain-relevant context (selector for element errors, param name for input errors).
- **Positive:** `#[source]` preserves the full error chain — compatible with `anyhow::Context` and `tracing`.
- **Positive:** `Cancelled` variant is semantically distinct — cancellation is not a "platform error".
- **Trade-off:** Slightly more verbose error construction. Factory methods (`ToolError::invalid_input(...)`, `ToolError::element_not_found(...)`) mitigate this.

## Decision 4: Mandatory CancellationToken propagation (§5.4)

### Problem

Before the refactor, some tools checked `CancellationToken` and others didn't. There was no consistent pattern.

### Solution

Every `Tool::execute()` method now:

1. Checks `token.is_cancelled()` at the start of execution.
2. Checks after any blocking operation (`spawn_blocking`, long-running UIA calls).
3. Returns `Err(ToolError::cancelled())` immediately on cancellation.

The `GraphExecutor` also checks cancellation:
- At the start of each node execution.
- At the start of each loop iteration.
- Before each retry attempt.

### Consequences

- **Positive:** Operations can be cancelled without killing the process — essential for long-running automation workflows.
- **Positive:** Consistent pattern across all 6 tools — any new tool follows the same contract.
- **Positive:** Cancellation is fast — no waiting for blocking operations to complete.
- **Trade-off:** Requires passing `CancellationToken` through the entire call chain.

## Decision 5: No unwrap/expect/panic! in production code

### Problem

Several `unwrap()` and `expect()` calls existed in non-test code — primarily in error branches that "should never happen":

```rust
let result = ctx.get("key").unwrap();  // Panics if key is missing
```

### Solution

All occurrences replaced with proper error propagation via `?` and `ToolError`. `unwrap()`/`expect()`/`panic!()` are now only allowed in:
- Test modules (`#[cfg(test)]`)
- `main()` for fatal errors (configuration missing, etc.)

### Consequences

- **Positive:** No unexpected panics in production — every error path returns a `Result`.
- **Positive:** The `#[cfg(test)]` boundary makes it obvious where panics can occur.
- **Trade-off:** Slightly more verbose error handling in some branches.

## Decision 6: Clippy and formatting enforcement

### Problem

No consistent linting — unused variables, redundant clones, and style inconsistencies accumulated over time.

### Solution

- `cargo clippy --all-targets` passes with zero warnings.
- `cargo fmt` enforced for consistent formatting.
- Key lints: `clippy::unwrap_used`, `clippy::expect_used`, `clippy::useless_format`, `clippy::collapsible_if`.

### Consequences

- **Positive:** CI can reject PRs that introduce warnings.
- **Positive:** Consistent code style reduces review friction.

---

## Consequences (summary)

1. **Type safety** — `Tool::Input`/`Tool::Output` replaces JSON `Value` at the type level. Compiler catches mismatches.
2. **State safety** — `ExecutionContext<Unvalidated>` → `ExecutionContext<Ready>` transition is enforced at compile time.
3. **Structured errors** — every `ToolError` carries domain-specific context (selector, param name, timestamp) and preserves the source chain.
4. **Graceful cancellation** — consistent `CancellationToken` checks at all logical boundaries across all tools.
5. **No panics** — `unwrap()`/`expect()`/`panic!()` only in test code.
6. **CI-ready** — zero clippy warnings, consistent formatting.
7. **Backward compatibility** — all existing tests (71) pass without modification.
