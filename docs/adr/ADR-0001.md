# ADR-0001: ToolRegistry, ElementSelector, and cross-platform strategy

**Status:** Accepted  
**Date:** 2026-07-02  
**Deciders:** @kurosss  

---

## Context

The smith-automation project requires architectural decisions in three areas:

1. **ToolRegistry** — how to register, discover, and execute tools.
2. **ElementSelector** — how to identify UI elements and bind them to processes.
3. **Cross-platform** — how to organize code to support Windows, Linux, and macOS.

---

## Decision 1: ToolRegistry — static registration for now

### Option considered: Static registration (chosen)

Tools are registered in a `HashMap<String, Box<dyn Tool>>` manually at the entry point:

```rust
let mut registry = ToolRegistry::new();
registry.register(ClickTool::new());
registry.register(TypeTool::new());
```

**Pros:**
- Simplicity of implementation at the current stage.
- Full control over the tool set at compile time.
- No unnecessary dependencies (dlopen, libloading).

**Cons:**
- Adding a new tool requires changing the core code.
- Cannot load external plugins without recompilation.

### Option considered: Dynamic loading (deferred)

Loading tools from dynamic libraries (`.dll`/`.so`) via `libloading`.

**Deferred because:**
- Complicates build and distribution.
- Requires stable ABI or C interface.
- Low need at the current stage (all tools are known).

### Decision

We use static registration. Dynamic loading will be considered when the need for external plugins arises.

## Decision 2: ElementSelector with PID binding

UI elements are identified through a combination of:

1. **Process PID** — restricts element search to a specific application.
2. **AutomationId / Name / ControlType** — standard UIA properties for precise identification.
3. **Condition tree** — support for compound conditions (AND, OR) via UIA `Condition`.

The element itself is wrapped in `SafeUIElement` — a thread-safe wrapper on `Arc<UIElement>` with `unsafe impl Send + Sync`:

- `UIElement` internally contains a COM interface which is free-threaded.
- All blocking COM calls (click, setValue) are moved into `spawn_blocking`.

**Decision:** `ElementSelector` will be a struct storing a PID and optional search conditions. After searching, the result (a found `SafeUIElement`) is stored in `ExecutionContext` for subsequent use by tools.

## Decision 3: Cross-platform strategy with cfg(windows)

### Levels of isolation

**Level 1 — Dependencies:**

```toml
[target.'cfg(windows)'.dependencies]
uiautomation = "0.25.0"
```

Platform-specific crates are only included for the corresponding target.

**Level 2 — Modules:**

```rust
#[cfg(windows)]
pub mod element;
#[cfg(windows)]
pub use element::SafeUIElement;
```

Modules and public types only exist on their respective platform.

**Level 3 — Implementation:**

```rust
#[cfg(windows)]
impl SomePlatformTrait for PlatformImpl { ... }
```

Individual impl blocks are hidden behind `cfg`.

### Future platform support

A separate crate is created for each new platform:

- `smith-windows` — UIAutomation (currently exists).
- `smith-linux` — AT-SPI / X11 (planned).
- `smith-macos` — Accessibility API (planned).

Active backend selection will be handled via feature flags or runtime registry registration.

### Decision

All platform-specific code is isolated behind `#[cfg(windows)]` at the module and dependency level. `smith-core` remains fully platform-independent. New platforms are added as separate crates with the same `Tool` interface.

---

## Consequences

1. **ToolRegistry:**
   - Simplicity at the cost of flexibility — tools are only part of the core.
   - Easy refactoring when transitioning to dynamic loading (registry API will not change).

2. **ElementSelector:**
   - PID binding isolates automation within a single process.
   - SafeUIElement requires `unsafe` — safety invariants are documented.
   - spawn_blocking ensures the async runtime is not blocked.

3. **Cross-platform:**
   - Each backend is a separate crate with minimal platform dependencies.
   - Compilation on an unsupported platform safely excludes irrelevant code.
   - Feature flags for backend selection will be added when a second platform appears.
